# JustDoThree v2.1 Backlog + Rollover Improvements

## Summary
Implement six product changes across rollover, backlog browsing, task creation, and completed-day task removal. The biggest behavior change is a new explicit chooser when multiple rollover tasks target a future day that does not have enough open primary-plan slots. The Plan/Today task-removal flow also gains a promotion rule: if a completed primary task is removed from a completed day and that day already has a stretch goal, promote the earliest stretch goal into the main three-task plan.

## Key Changes
- Rollover overflow selection
  - Extend the rollover scheduling flow so `scheduleFor(date)` is no longer “best effort” when the destination day is short on slots.
  - In the rollover UI, if multiple rollover items are assigned to the same future day and their count exceeds that day’s available primary slots, present an explicit chooser for that date group before confirm succeeds.
  - The chooser must let the user decide which tasks keep the available slots and which task(s) go to backlog.
  - After the user resolves that overflow, the final choices applied to `RolloverEngine` should be decision-complete: selected tasks remain `scheduleFor(date)`, overflow tasks become `.backlog`.
  - `RolloverEngine.applyChoices()` should continue to be deterministic and should not silently drop scheduled tasks when the target day is full; the UI should resolve conflicts first.

- Backlog search
  - Add search to the main backlog screen using a persistent search field.
  - Filter only the visible backlog rows; keep current sort order semantics for the filtered result.
  - Search should be case-insensitive substring matching on task title.
  - Reordering should only remain enabled when no search query is active, so drag-and-drop cannot reorder a filtered subset incorrectly.

- Future-day note in backlog list
  - Add a small secondary note to `BacklogRow` when a task is already planned on a future day.
  - Use the same visual tier as the existing rollover/recurring metadata: a small line under the title, not a badge in the main row.
  - Show one note only, using the nearest future planned day if a task appears in multiple future plans.
  - Keep the current add-to-plan picker behavior unchanged; it already shows future scheduling separately at the bottom.

- Paste support in `AddTaskSheet`
  - In create mode only, add a second import action alongside file import inside the single-task add sheet.
  - Reuse the existing paste-import experience and duplicate handling rather than inventing a second parser.
  - Best implementation path: present `PasteTasksSheet` from `AddTaskSheet` just like the existing import instructions/file flow, and keep edit mode unchanged.

- New tasks at top of backlog
  - Change all task-creation/import entry points so newly added tasks appear at the top of backlog rather than the bottom.
  - This includes:
    - `AddTaskSheet` single create
    - `BacklogPickerSheet` inline create
    - backlog file import
    - backlog paste import
    - add-sheet file import
    - new add-sheet paste import
  - Normalize sort insertion by assigning new items sort orders smaller than the current minimum, preserving the input order of multi-adds so the first pasted/imported line appears highest among the newly added block.
  - Leave manual drag reordering behavior unchanged after insertion.

- Promote stretch goal when removing from a completed day
  - When a user removes a completed primary task from a day whose three primary tasks are all complete, and that day has at least one stretch goal, automatically promote one stretch goal into the primary `taskIDs`.
  - Promote the first stretch goal in that day’s existing `stretchTaskIDs` order.
  - Remove the promoted task from `stretchTaskIDs` and, if it had already been completed as a stretch goal, move its completion state from `completedStretchIDs` to `completedTaskIDs` so the day stays complete.
  - Apply this behavior anywhere the app removes a primary task from a non-today day plan, including the Plan screen’s remove/replace flow.
  - Do not promote anything if the removed task was not completed, the day was not fully complete, or there are no stretch goals available.

## Interfaces / Behavior Changes
- `RolloverSheet` gains a conflict-resolution subflow for grouped future-day scheduling overflow before confirm.
- `BacklogView` gains searchable state and disables move/reorder while a search is active.
- `BacklogRow` needs access to planned-day metadata, likely by passing an optional `futurePlanLabel` string from `BacklogView`.
- `AddTaskSheet` gains a paste-import entry point in create mode.
- Shared task insertion/import logic should be centralized or mirrored consistently so all “new task” paths insert at the top using the same sort-order strategy.
- Planner task removal logic needs a reusable promotion helper in the plan engine layer so UI screens do not hand-roll the stretch-to-primary promotion rule.

## Test Plan
- Rollover
  - Three rollover tasks scheduled to a future day with 3 open slots: all schedule successfully, no chooser.
  - Three rollover tasks scheduled to a future day with 2 open slots: chooser appears, user picks 2 to keep and 1 goes to backlog.
  - Same overflow case after one item was individually set already: chooser still reflects final grouped state correctly.
  - Confirm cannot silently drop an overflow task; final applied choices match the chooser result.
- Backlog search
  - Search filters rows by title case-insensitively.
  - Empty query restores the full backlog.
  - Reordering is unavailable while searching and works again when search is cleared.
- Future-day note
  - A backlog task planned for tomorrow shows “Planned for …” beneath the title.
  - A task planned for today does not show the future-day note in backlog.
  - A task planned on multiple future days shows only the nearest upcoming day.
- Add/paste/import
  - `AddTaskSheet` create mode offers both file import and paste import.
  - Edit mode does not show paste/file import actions.
  - Pasted/imported duplicates are skipped exactly like current behavior.
- Top insertion
  - Single created task appears at the top of backlog.
  - Multi-line paste/import adds a contiguous block at the top in the same order entered.
  - Inline create from plan/today picker also lands at the top of backlog if later viewed there.
- Stretch promotion on removal
  - Removing a completed primary task from a fully completed day with stretch tasks promotes the first stretch goal into the main three.
  - If that promoted stretch task was already completed, the day remains fully complete after promotion.
  - Removing a non-completed primary task does not promote anything.
  - Removing from a day with no stretch goals leaves the slot open as it does today.
  - Replacing a completed primary task on a completed day triggers the same promotion logic before/after replacement in a way that keeps plan state consistent.

## Assumptions
- “Next day” in the rollover request should be implemented generically for any scheduled future day, not only tomorrow.
- The backlog future-day note belongs only in the main backlog list; the picker remains as-is.
- Newly inserted tasks should go above all existing backlog items, not above pinned/completed/today items, since those are already excluded from the backlog list.
- For stretch promotion, the promoted item is chosen by current stretch ordering, not by newest or completed-first heuristics.
- Item numbering typo in the request (`6`) is treated as the fifth requested change, not a missing extra requirement.
